Целью лабораторной работы является знакомство с системой управления версиями SVN и системой ведения проекта Trac.

В ходе выполнения лабораторной работы студенты должны приобрести навыки работы

  1. с консольным клиентом svn,
  2. с системой отслеживания ошибок и документирования в Trac.

1. Условие задачи ¶

Исходными данными для выполнения лабораторной работы является исходный код программы, которая решает некоторую задачу. Программа состоит из модуля, реализующего основные функции и/или процедуры, и основной программы. Основная программа содержит процедуру, которая формирует массив с тестовыми данными, и вызов функций и/или процедур из модуля. Программа компилируется и успешно работает на простых тестовых данных, т.е. находится в том состоянии, когда ее можно поместить под версионный контроль.

В программе допущена ошибка. Эту ошибку вам надо найти и исправить.

Допущенная в программе ошибка не связана с вводом некорректных данных или с передачей в функции и/или процедуры массивов нулевой длины. При поиске ошибки отталкивайтесь от предложенной реализации основной функции и/или процедуры.

Выполнение лабораторной работы включает следующие крупные шаги:

  1. Получение локальной копии репозитория; помещение исходной программы под версионный контроль.
  2. Разработку теста, для которого задача решается неверно; добавление теста к программе; фиксацию изменений.
  3. Составление отчета об ошибке.
  4. Исправление ошибки; фиксацию изменений; закрытие отчета об ошибке.
  5. Работа с wiki.

Замечания.

  1. Ниже по тексту, строки, которые начинаются с символа '#', являются либо пояснениями, либо примерами выдачи той или иной команды.
  2. Команды, приведенные в данном руководстве, нужно набирать в консоли самостоятельно (!), а не копировать из руководства! Скопированную команду svn-клиент может понять неправильно из-за проблем с кодировкой.
  3. Комментарии к фиксациям изменений нужно писать на английском языке и при этом стараться делать их максимально понятными, но короткими.
  4. Приводимые в примерах имена файлов могут отличаться от имен файлов в выданном вам исходном коде.

2. Создание локальной копии репозитория ¶

2.1
Запустите консольный режим: Для этого нажмите клавиши WIN+R и в открывшемся диалоговом окне наберите команду cmd. Находясь в консольном окне, перейдите на диск D: в папку temp.

Команды, приведенные ниже, нужно набирать в окне консоли. Обязательно читайте вывод команды! 

# Изменим размеры окна консоли
mode con lines=40 cols=120

# Выберем диск D
d:

# Перейдем в каталог temp с помощью команды cd (change directory). Если такого каталога нет, создадим его с помощью команды mkdir temp.
cd temp

# При успехе должно появиться
# D:\temp>

# Посмотрим содержимое каталога temp с помощью команды dir
dir

# дата <DIR> .
# дата <DIR> ..
# возможно что-то еще

# .  - "короткое имя" текущего каталога, в нашем случае D:\temp
# .. - "короткое имя" родительского каталога, в нашем случае D:\


2.2
Создайте локальную (рабочую) копию репозитория и перейдите в каталог репозитория.

Для получения локальной копии репозитория вам понадобится ИМЯ_ПРОЕКТА, ВАШЕ_ИМЯ_ПОЛЬЗОВАТЕЛЯ и пароль. Эта информация должна была придти вам по электронной почте.

В системе dev.iu7 ИМЯ_ПРОЕКТА по курсу "Основы программной инженерии" образуется по следующему правилу: 

se_15_letter-surname
  • letter - первая буква имени;
  • surname - фамилия.

ИМЯ_ПОЛЬЗОВАТЕЛЯ образуется по следующему правилу 

surname_initial_email
  • surname - фамилия;
  • initial - инициалы;
  • email - часть адреса электронной почты до символа @.

Для записи фамилии, инициалов и первой буквы имени используется транслитерация. Обратите внимание на использование символов "подчеркивание" и "тире".

Замечание.
Объяснение правил, по которым формируется имя проекта и имя пользователя, приводится с целью облегчить запоминание этих имен. Эти имена не нужно "создавать" самостоятельно, они уже зарегистрированы в системе dev.iu7. 

svn checkout http://dev.iu7.bmstu.ru/svn/ИМЯ_ПРОЕКТА --username ВАШЕ_ИМЯ_ПОЛЬЗОВАТЕЛЯ

# Вас попросят ввести пароль

# При успехе должно появиться
# Checked out revision 0.

# Проверим, что в каталоге temp появилась локальная копия репозитория
dir se_*

# дата <DIR> ИМЯ_ПРОЕКТА

cd ИМЯ_ПРОЕКТА

Каталог ИМЯ_ПРОЕКТА не пустой. Чтобы убедиться в этом наберите команду 

# Вывод файлов/каталогов с атрибутом "скрытый"
dir /AH

Скрытый каталог .svn содержит служебные файлы системы контроля версий. Никогда сами не удаляйте этот каталог! В противном случае вы испортите вашу локальную копию, что повлечет за собой необходимость ее повторной выгрузки, но уже в другой каталог, а также внесения в дублирующую версию рабочей копии всех тех изменений, которые уже были вами сделаны, но не были зафиксированы на сервере.

Замечания.

  1. Для получения помощи по той или иной "подкоманде" (subcommand) достаточно написать svn help subcommand.
  2. Существует сокращенная форма "подкоманды" checkout - co, т.е. вместо svn checkout можно писать svn co.

3. Создание структуры проекта ¶

В "классической" структуре на каждый проект создаются три каталога: trunk (основная ветвь), branches (ветви) и tags. 

project\
       |- trunk\
       |- branches\
       |- tags\

В каталоге trunk хранится исходный код приложения. Он всегда поддерживается в рабочем состоянии и представляет собой последнюю стабильную версию приложения.

Если в какой-то момент необходимо добавить новую функциональность, то создается ветвь в папке branches и разработчики могут спокойно модифицировать копию в branches, не опасаясь, что основная версия будет затронута. Когда новая функциональность готова, выполняется слияние ее с основной версией.

Перед выпуском очередной версии (релиза) программного продукта недопустимо вносить серьезные изменения в исходный код. Поэтому перед выпуском обычно создаётся релизная ветвь (tag), изменения в которой строго регламентированы. В основном в неё попадают исправления серьёзных ошибок, обнаруженных при подготовке версии.

На лабораторных работах мы будем использовать упрощенную структуру: один каталог на один проект.


Перейдите в каталог ИМЯ_ПРОЕКТА, создайте в нем папку lab_01 и добавьте ее под версионный контроль. 

mkdir lab_01

# Проверим состояние репозитория
svn status

# Должно появиться 
# ?       lab_01

# Некоторые значения статуса, которые могут принимать файлы или каталоги:
#   A — элемент был запланирован для добавления в репозиторий ("Added");
#   C — элемент находится в состоянии конфликта ("Conflict");
#   D — элемент запланирован для удаления из репозитория ("Deleted");
#   M — элемент был изменен ("Modified");
#   ? — элемент не находится под версионным контролем;
#   ! — элемент находится под версионным контролем, но отсутствует или поврежден (например, был удален с помощью не-svn команды).

# Добавим каталог под версионный контроль
svn add lab_01

# При успехе должно появиться
# A       lab_01

# Зафиксируем изменения
svn commit –m "Folder of first exercise was created." --username ВАШЕ_ИМЯ_ПОЛЬЗОВАТЕЛЯ

# При успехе SVN выведет ряд сообщений, в конце:
# Commited revision 1.

Замечание.
Существует сокращенная форма "подкоманды" commit - ci, т.е. вместо svn commit можно писать svn ci.

4. Добавление файлов в проект ¶

4.1
Загрузите исходный код программы. Программа решает следующую задачу: дан целочисленный массив, необходимо найти положение максимального элемента.

Распакуйте архив, файлы из него поместите в папку lab_01.

4.2
Убедитесь в том, что программа компилируется и работает на тестовом примере. Обязательно поместите исходные коды программы под версионный контроль.

В репозитории следует хранить только исходные тексты (а не исполняемые или объектные файлы). Обычно все, что помещено в репозиторий, должно компилироваться и проходить тесты. 

# Прежде чем работать в командной строке выйдите из Lazarus.

cd lab_01

svn status

# Должно появиться примерно следующее
# ?       main.lpr
# ?       main.lps
# ?       main.exe
# ?       lib
# ?       main.lpi
# ?       iarray.pas

# Файлы программы в вашем варианте могут называться по-другому.
svn add iarray.pas main.lpr main.lpi main.lps

svn status

# Если все сделано правильно, вы должны увидеть
# A       main.lpr
# A       main.lps
# ?       main.exe
# ?       lib
# A       main.lpi
# A       iarray.pas

svn commit –m "Initial version of program was added." --username ВАШЕ_ИМЯ_ПОЛЬЗОВАТЕЛЯ

Замечание.
Существует сокращенная форма "подкоманды" status - st, т.е. вместо svn status можно писать svn st.

5. Внесение изменений ¶

5.1
Найдите тест, на котором программа будет работать неверно. Оформите его как процедуру, которая должна иметь следующий заголовок 

procedure Test2(var Arr : TIArray; var N : integer)

Добавьте вызов новой тестовой процедуры (вызов старой тестовой процедуры оставьте, потому что количество тестов должно только увеличиваться). Сохраните ваши изменения и выйдите из Lazarus.

Замечание.
На данном этапе необходимо только добавить в программу тест, демонстрирующий, что основной алгоритм работает неверно. Исправлять сам программный код не нужно.

Проанализировав изменения с помощью команды svn status, вы увидите примерно следующее 

M       main.lpr
M       main.lps
?       main.exe
?       backup
?       lib

Файл main.lpr изменили вы сами, а файл main.lps – Lazarus.

Назначение некоторых файлов Lazarus:

РасширениеАкронимНазначение
lpiLazarus Project InformationСодержит настройки проекта (например, директивы компиляции).
lpsLazarus Project SessionСодержит индивидуальные настройки (например, точки останова).
lprLazarus PRogramТочка входа

Файл *.lps изменяется самой Lazarus, и помещение его под версионный контроль было ошибкой. Чтобы не фиксировать изменения в этом файле, отменим их. Для отмены изменений, которые еще не отправлены на сервер, используется команда svn revert. 

svn revert main.lps

Reverted 'main.lps'

Замечание.
В зависимости от ваших действий файл lps мог и не измениться.

Чуть позже мы удалим этот файл из-под версионного контроля.

Теперь проанализируем изменения с помощью команды svn diff. В отличие от команды svn status команда svn diff показывает не перечень файлов или каталогов, а сами изменения, то есть то, что изменил пользователь в каждом конкретном файле. 

svn diff

Index: main.lpr
===================================================================
--- main.lpr    (revision 2)
+++ main.lpr    (working copy)
@@ -16,6 +16,16 @@
   Arr[4] := 8;
 end;

+procedure Test2(var Arr : TRArray; var N : integer);
+begin
+  N := 4;
+
+  Arr[1] := -1;
+  Arr[2] := -2;
+  Arr[3] := -3;
+  Arr[4] := -4;
+end;
+
 var
   Arr : TRArray;
   N    : integer;
@@ -24,5 +34,8 @@
   Test1(Arr, N);
   writeln('Max = ', GetMax(Arr, N):8:4);

+  Test2(Arr, N);
+  writeln('Max = ', GetMax(Arr, N):8:4);
+
   readln;
 end.

Давайте подробнее разберем формат вывода команды diff. Первое, что показывает команда, - это имя файла и номера ревизий, которые сравниваются: 

Index: main.lpr
===================================================================
--- main.lpr    (revision 2)
+++ main.lpr    (working copy)

Для файла main.lpr мы сравниваем ревизию 2 (это последняя ревизия на данный момент) и рабочий файл, который хранится на локальном компьютере.

Следующая строка содержит данные о том, какие строки выведены ниже: 

@@ -16,6 +16,16 @@

-16,6 означает "в количестве 6 строк, начиная со строки 16", а минус в начале говорит о том, что это относится к файлу ревизии 2, который обозначается минусами: 

--- main.lpr    (revision 2)

Далее идут аналогичные метки строк, но для рабочего файла: +16,16 — начиная с 16 строки, вывелись 16 строк.

Ниже выведен листинг, охватывающий несколько неизмененных строк, после которых располагаются изменения. 

   Arr[4] := 8;
 end;

+procedure Test2(var Arr : TRArray; var N : integer);
+begin
+  N := 4;
+
+  Arr[1] := -1;
+  Arr[2] := -2;
+  Arr[3] := -3;
+  Arr[4] := -4;
+end;
+
 var
   Arr : TRArray;
   N    : integer;

Система SVN считает все изменения целой строкой. Это означает, что если вы исправили в строке хотя бы одну букву, то SVN расценит это как удаление старой строки и вставку новой.

Зафиксируйте ваши изменения командой svn commit.

5.2
Создайте в Trac отчет об ошибке.

Для этого авторизуйтесь на  http://dev.iu7.bmstu.ru, выберите вкладку "Проекты", затем "Показать" (это достаточно сделать один раз) и, наконец, ссылку на ваш репозиторий Trac ("trac"). После повторной авторизации (ссылка "login") вы попадете на страницу вашего проекта. Она будет выглядеть как показано на рисунке

Выберите вкладку "New Ticket". На открывшейся странице заполните указанные поля.

В поле описания ошибки (Description) можно использовать команды форматирования Wiki. В приводимом примере их две: "!" - отменяет интерпретацию следующего выражения как команды Wiki; "[число]" - используется для указания номера ревизии (в данном случае - это ревизия, в которой найдена ошибка).

Создайте отчет, нажав "Create ticket".

Вы можете просмотреть отчеты об ошибках, которые есть в вашем проекте, выбрав "View Tickets", а затем, например, запрос "My tickets".

При просмотре только что созданной ошибки обязательно щелкните по номеру ревизии. Вы увидите изменения, которые были сделаны в этой ревизии, но уже в "графическом" представлении.

5.3
Удалите файл main.lps из-под версионного контроля. 

svn del main.lps

svn status

svn commit -m "lps was deleted."

5.4
Исправьте ошибку.

Прежде чем приступить к исправлению ошибки измените ее состояние с "Новой" (New) на "В работе" (Accepted). Для этого откройте отчет об ошибке, выберите одну из ссылок "Modify" (она сверху) или "Modify ticket" (она внизу), в появившейся группе "Action" выберите переключатель "accept", а затем нажмите на "Submit changes".

Прежде чем фиксировать ваше исправление в репозитории проанализируйте сделанные изменения с помощью команд svn status, svn diff.

После фиксации изменений закройте ошибку ("resolve as fixed"). В комментарии не забудьте указать номер ревизии, в которой ошибка была исправлена.

6. Анализ истории ¶

Проанализируйте историю изменений с помощью команды svn log.

С помощью параметра -r можно задать номера ревизий, которые вас интересуют. Можно указывать как одну ревизию, так и диапазон значений. 

svn log -r 1:HEAD

HEAD - предопределенное значение, которое обозначает последнюю ревизию. Для выдачи подробной информации о фиксации существует параметр -v.

С помощью команды svn diff можно анализировать изменения между любыми фиксациями. В качестве аргументов этой команде можно передавать имя файла и/или ревизии, которые будут сравниваться.

  • svn diff - сравнить рабочие файлы с последней ревизией;
  • svn diff main.lpr - сравнить локальный файл main.lpr с последней ревизией этого файла в репозитории;
  • svn diff -r 3 - сравнить все локальные файлы с ревизией номер 3;
  • svn diff -r 4 main.lpr - сравнить локальный файл main.lpr с ревизией номер 4;
  • svn diff -r 1:4 main.lpr - сравнить файл main.lpr ревизии 1 с этим же файлом, но ревизии 4 в репозитории;
  • svn diff -r 1:4 - просмотр отличий между любыми фиксациями.

Изменения между ревизиями можно посмотреть и в Trac. Для этого нужно зайти на страницу вашего проекта и выбрать вкладку "Browse Source".

7. Работа с Wiki ¶

Краткое руководство по Wiki

Создайте в Wiki отдельную страницу. Ссылку на эту страницу разместите на первой странице Wiki.

На созданной странице разместите несколько небольших абзацев текста с различным оформлением заголовков и самого текста. Текст обязательно должен включать в себя список и рисунок.

9. Защита лабораторной работы ¶

Для защиты лабораторной работы необходимо подготовить отчет. В отчете необходимо привести пошаговое описание выполнения лабораторной работы для индивидуальной задачи.

Описание каждого шага включает

  • описание назначения шага;
  • команду svn (команда приводится полностью со всеми параметрами);
  • вывод этой команды.

Номер задачи выбирается в зависимости от вашего номера в журнале: нечетные номера делают задачу 1 (подсчитать сколько раз в массиве встречается максимальный элемент), четные – задачу 2 (сформировать новый массив на основе заданного).

Отчет оформляется на отдельной странице wiki.
